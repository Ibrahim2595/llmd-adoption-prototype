This scheduler makes optimized routing decisions for inference requests to
the llm-d inference framework.

## About

This provides an "Endpoint Picker (EPP)" component to the llm-d inference
framework which schedules incoming inference requests to the platform via a
[Kubernetes](https://kubernetes.io) Gateway according to scheduler plugins. For more details on the
llm-d inference scheduler architecture, routing logic, and different plugins
(filters and scorers), including plugin configuration, see the [Architecture Documentation](https://github.com/llm-d/llm-d-inference-scheduler/blob/main/docs/architecture.md).

### Relation to GIE (IGW)

The EPP extends the [Gateway API Inference Extension (GIE)](https://github.com/kubernetes-sigs/gateway-api-inference-extension) project,
which provides the API resources and machinery for scheduling. We add some
custom features that are specific to llm-d here, such as [P/D Disaggregation](https://github.com/llm-d/llm-d-inference-scheduler/blob/main/docs/disagg_pd.md).
The two projects collaborate closely as often a feature in llm-d might require
enablement and extensions in the GIE code base.
Unique and experimental features may start in llm-d and migrate, over time, to
GIE. As a project goal, we prefer to upstream functionality to GIE when
- it has matured sufficiently and has proven wide applicability and usefulness; and
- it can be implemented in EPP alone (i.e., llm-d provides a full inference framework,
  beyond scheduling).

Note that in general features should go to the upstream [Gateway API Inference
Extension (GIE)](https://github.com/kubernetes-sigs/gateway-api-inference-extension) project first if applicable. The GIE is a major dependency of
ours, and where most general purpose inference features live. If you have
something that you feel is general purpose or use, it probably should go to the
GIE. If you have something that is llm-d specific then it should go here. If
you are not sure whether your feature belongs here or in the GIE, feel free to
create a [discussion](https://github.com/llm-d/llm-d-inference-scheduler/discussions/new?category=q-a) or ask on [Slack](https://llm-d.slack.com/).

A compatible [Gateway API](https://github.com/kubernetes-sigs/gateway-api) implementation is used as the Gateway. The Gateway
API implementation must utilize [Envoy](https://github.com/envoyproxy/envoy) and support [ext-proc](https://www.envoyproxy.io/docs/envoy/latest/configuration/http/http_filters/ext_proc_filter), as this is the
callback mechanism the EPP relies on to make routing decisions to model serving
workloads currently.

## Contributing

Our community meeting is bi-weekly at Wednesday 10AM PDT ([Google Meet](https://meet.google.com/uin-yncz-rvg), [Meeting Notes](https://docs.google.com/document/d/1Pf3x7ZM8nNpU56nt6CzePAOmFZ24NXDeXyaYb565Wq4)).

maintainers can do an assessment, and work on the details with you.
